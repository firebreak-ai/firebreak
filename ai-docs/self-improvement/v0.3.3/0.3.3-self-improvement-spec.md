# 0.3.3 Self-Improvement: Detection Scope Expansion and Pipeline Hardening

## Problem

The Firebreak code review pipeline's detection scope is shaped by its AI failure mode taxonomy. This taxonomy effectively catches how AI-generated code specifically fails — semantic drift, dead infrastructure, test integrity decay, copy-paste duplication — but creates a blind spot for standard engineering issues that any competent reviewer would flag regardless of code origin.

A post-remediation full-codebase review of a brownfield project found 60 verified findings. 55% (33 findings) were in code that was never in scope for any per-phase review, and many of those were standard engineering concerns (context propagation, string-based error dispatch, sentinel value confusion, goroutine lifecycle) outside the current taxonomy. Additionally, the review exposed operational gaps: no severity tiering in findings, no cross-cutting pattern grouping, no stuck-agent recovery, and test-integrity blind spots in the pipeline's own output.

This self-improvement cycle applies 54 proposals from two sources: the Phase 6 remediation retrospective (11 proposals) and the post-remediation full-codebase code review (43 proposals from 8 improvement analysts).

## Goals / Non-goals

**Goals:**
- Expand the AI failure mode checklist and structural detection targets to cover standard engineering review concerns
- Add severity tiering to the Detector/Challenger output so critical behavioral bugs are distinguishable from structural debt
- Harden the code review skill's operational model (stuck-agent recovery, parallel spawning, cross-unit pattern grouping)
- Close the test reviewer's blind spots for non-enforcing tests, stale annotations, and advisory assertions
- Improve implementation guide and task compilation with observed friction-reduction rules

**Non-goals:**
- Rewriting or restructuring existing assets beyond targeted additions
- Adding language-specific detection items beyond the Go-specific patterns observed (context bypass, build tags) — these are included as examples but the items should be language-neutral where possible
- Automating any of these checks as hook scripts or gate validations — this cycle adds instructions and detection targets only
- Changing the Detector/Challenger architecture — the adversarial pattern is validated; this cycle expands what it looks for, not how it works

## User-facing behavior

The user interacts with these changes through the existing Firebreak skills:

- `/fbk-code-review`: Detectors produce findings with severity tiers and cross-cutting pattern labels. Challengers can report adjacent observations. The orchestrator groups findings by cross-cutting pattern and presents behavioral bugs first. Stuck agents trigger a recovery protocol instead of silent fallback.
- `/fbk-implement`: Hook rejection loops are capped at 3 retries. Fresh agents per task (no worker reuse). Verification and hooks run in foreground.
- `/fbk-breakdown`: E2E harness tasks are combined. Multi-site tasks have per-site completion conditions.
- Test reviewer: Catches stale failure annotations, empty gate tests, advisory assertions, unconditionally skipped behavioral tests, and phantom assertion strings.

## Technical approach

All changes are additive text modifications to existing Firebreak context assets (markdown instruction files, agent definitions). No code, no hooks, no gate scripts.

**AC numbering note:** AC numbers correspond to original proposal IDs from the self-improvement input document. Gaps (AC-04, AC-19, AC-20, AC-30) indicate rejected proposals.

### 4.2 Finding classification and review output

Finding classification uses two orthogonal axes, aligned with industry consensus (SonarQube, CodeClimate, CodeRabbit all use independent type and severity dimensions). Canonical definitions live in `code-review-guide.md` (per OQ-2).

**Type axis** (what kind of issue — assigned by Detector):
- `behavioral` — code does something different from what its name, documentation, or spec says
- `structural` — code organization issue (duplication, dead code, dead infrastructure, bare literals)
- `test-integrity` — test provides less coverage than it appears to
- `fragile` — code works now but breaks under likely future changes (string-based dispatch, sentinel confusion, context bypass)

**Severity axis** (how urgent — validated by Challenger):
- `critical` — affects production correctness, security, or data integrity now
- `major` — significant risk under realistic conditions
- `minor` — should be addressed but not urgently
- `info` — informational; accurate observation but no action required

Nits remain excluded from findings entirely (per AC-26) and do not receive severity classification. The Challenger's existing "downgrade to nit" behavior becomes "reject as nit" — the sighting is excluded from the verified findings list and counted separately.

**Type axis disambiguation rule:** When an issue fits multiple types, classify by the primary risk. If the code produces wrong results now, it is `behavioral`. If it produces correct results but will break under realistic changes, it is `fragile`. If it is a code organization problem with no correctness risk, it is `structural`. If a test provides less coverage than it appears to, it is `test-integrity`.

### Category-to-type migration

The existing system uses a single `category` field with values `semantic-drift`, `structural`, `test-integrity`, `nit`. This is replaced by two orthogonal axes: type and severity. The migration:

- `semantic-drift` → type `behavioral`
- `structural` → type `structural` (unchanged)
- `test-integrity` → type `test-integrity` (unchanged)
- `nit` → removed from type axis; nits are excluded from findings per AC-26
- `fragile` → new type value (no prior equivalent)
- Severity axis (`critical`, `major`, `minor`, `info`) → entirely new

Assets requiring migration:
- `code-review-guide.md`: sighting format template (replace `Category` field with `Type` + `Severity`), finding format template (same), Category Values section (replace with canonical two-axis definitions)
- `fbk-code-review-detector.md`: output instruction (replace `category` with `type` + `severity`)
- `fbk-code-review-challenger.md`: validation instruction (validate both axes), nit-downgrade behavior (replace "downgrade to nit" with "reject as nit — exclude from findings, count separately")

**Detector agent (`fbk-code-review-detector.md`)** — 3 modifications:
- Remove Bash from tools list and delete linter discovery section (enforce read-only per authoring rules); relocate linter discovery/execution to SKILL.md orchestration instructions (orchestrator runs linters before spawning Detectors and includes results in review context)
- Add type and severity fields to sighting output schema (type from the 4 types above; severity as initial estimate)
- Add cross-cutting pattern label field to sighting output schema

**Challenger agent (`fbk-code-review-challenger.md`)** — 4 modifications:

- Add cross-reference caller tracing requirement for `behavioral` type sightings
- Add `verified-pending-execution` status for `test-integrity` type sightings requiring test execution

### Implementation note: conventions for implementing agents

- **Agent definition frontmatter**: Detector and Challenger use YAML frontmatter (`name`, `description`, `tools`, `model`). When modifying the tools list or output schema, also update the `description` field to reflect the changes so agent definitions remain self-documenting.
- **Detection source values**: The allowed values (`spec-ac`, `checklist`, `structural-target`) are canonically defined in `code-review-guide.md` (sighting format section). New detection source values (e.g., `linter`) must be added there, not invented in SKILL.md spawn instructions.

### Implementation note: `code-review-guide.md` density

This file is 94 lines and receives 8 ACs, roughly doubling its size. New sections must be added as clearly headed subsections under the existing structure, not appended as a wall of text. The task compiler must structure additions to preserve scannability.

**OQ-2** — Single canonical definition or duplicated? Resolved: canonical two-axis classification (type × severity) in `code-review-guide.md`, referenced from Detector, Challenger, and `existing-code-review.md`. Aligns with industry consensus (SonarQube, CodeClimate, CodeRabbit).

**OQ-3** — Selective Challenger allocation? Resolved: rejected. AC-19 dropped. The supervisor bypass incident demonstrated a 53% miss rate when verification was skipped. Risk outweighs token cost.

**OQ-4** — Language-specific or language-neutral checklist items? Resolved: language-neutral with examples only where the principle alone would be ambiguous. Avoid over-tuning.

**OQ-5** — E2E test quality guidance? Resolved: out of scope for task compilation. Covered by existing ACs (AC-07 assertion specificity, AC-08 test name accuracy, AC-51 advisory assertion detection).

**OQ-6** — Severity taxonomy research? Resolved: industry uniformly separates type from severity as orthogonal axes. Original 4-tier system conflated them. Replaced with two-axis model: type (behavioral, structural, test-integrity, fragile) × severity (critical, major, minor, info).
