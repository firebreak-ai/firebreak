# v0.3.4 Self-Improvement Report

**Source**: 5 retrospectives from the v0.3.4 remediation cycle (Phase 0 behavioral bugs, Remediation 2 Phases 1-3, Intermediate retrospective)
**Assets analyzed**: 34 analysts across all Firebreak skills, agents, and fbk-docs
**Total raw proposals**: ~95 (before deduplication)

## Key Findings

The retrospectives reveal **three dominant failure patterns** that recurred across multiple phases:

1. **Compilation gaps** (every phase): Task compilers and spec authors enumerate only the callers the spec lists, missing secondary call sites. Every phase had at least one build failure from an unscoped caller.
2. **Skipped final verification** (Phase 1, Phase 2): The orchestrator substituted a build/compile check for the test suite despite documented corrective actions. Build tools in many languages don't compile test files, so test-file failures are invisible.
3. **Semantically incoherent test fixtures** (Phase 0): Agents followed fixture instructions literally, producing internally contradictory test scenarios that pass because mocks don't validate consistency.

Three secondary patterns also emerged:
4. **Dead conditional guards after removal** (Phase 0): Agents remove direct references but leave downstream guards unreachable.
5. **Placeholder values surviving to production** (Phase 2): Task compilers translate spec intent into literal placeholders (`context.Background()`) that agents implement verbatim.
6. **Incomplete extraction leaving residual duplication** (Phase 3): Agents extract shared logic but leave identical construction at call sites.

## Consolidated Proposals

Proposals are grouped by theme. Each proposal has a unique number for selection. Many themes touch multiple assets — the same principle must be reinforced at each pipeline stage to close the gap.

### Theme A: Full Test Suite Verification Hard Gate (revised)

**Root cause**: The orchestrator substituted a build/compile check for the test suite in two consecutive phases. Build tools in many languages do not compile test files. A documented corrective action did not prevent recurrence.

**Retro source**: Phase 1 Process Gaps, Phase 2 Process Gaps, Intermediate "needs a hard gate"

**Strategy**: Define "full test suite" by its *output requirement* (per-test pass/fail results), making build-check substitution logically impossible without naming any language-specific command. One primary definition in the implementation guide; secondary reinforcement at other pipeline stages.

## Issues Discovered

### 1. `fbk-brownfield-spec.md` has no explicit routing

**Confirmed**: No skill or routing table loads `fbk-brownfield-spec.md` by path. Neither `fbk-spec/SKILL.md` nor `feature-spec-guide.md` reference it. The feature-spec-guide has inline brownfield content (section 5b) but doesn't route to the separate file. Its sibling `fbk-brownfield-breakdown.md` IS explicitly routed from `fbk-breakdown/SKILL.md`.

If agents find `fbk-brownfield-spec.md` during code review, it's through codebase search (exploring `fbk-docs/`), not through a load instruction. This is fragile — it depends on agent exploration behavior rather than deterministic routing.

Proposals 18-19 improve the file's content but have no effect until a routing fix is applied to `fbk-spec/SKILL.md` or `feature-spec-guide.md`.

### 2. Reference integrity test has a blind spot for top-level `fbk-docs/` files

The orphan scan in `test-reference-integrity.sh` (Part 1, lines 59-67) uses `find -mindepth 2` from `fbk-docs/`, which only scans files inside subdirectories (e.g., `fbk-docs/fbk-context-assets/agents.md`). Top-level files like `fbk-brownfield-spec.md` and `fbk-brownfield-breakdown.md` sit at depth 1 and are excluded — the test treats them as index docs. Neither is checked for orphan status.

This gap allowed the missing routing for `fbk-brownfield-spec.md` to go undetected. The test should either: (a) include depth-1 `fbk-docs/fbk-*.md` files in the orphan scan, or (b) explicitly classify which top-level files are indexes vs. standalone docs and check the standalone ones.

### 3. 4 of 6 council agent files not analyzed

Transient issue — files are readable (644 permissions). The improvement analyst could not read `fbk-council-builder.md`, `fbk-council-guardian.md`, `fbk-council-security.md`, or `fbk-council-advocate.md` during its session. Proposals 86-97 cover only the architect and analyst agents; the other 4 council agents need a separate analysis pass.
