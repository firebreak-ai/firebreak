# 0.5.0 Design Discussion: State Machine Orchestration

Status: **brainstorming** (not a spec)

## Core thesis

Firebreak should be software that uses LLMs as compute units, not a collection of prose instructions that hopes LLMs follow them. A state machine + code-driven orchestrator moves control authority from prompt compliance to deterministic code.

## What this solves

The current architecture relies on LLMs to: write artifacts in the right format, call gates at the right time, respect stage ordering, pass correct context to subagents, and stop when they should. This works but doesn't scale to autonomous operation, and every instruction-following failure is silent until it causes a downstream problem.

## Constraints that shape the design

- **Context is append-only.** Agents can't selectively prune history. Everything accumulates until lossy compaction. Root cause of context rot.
- **Hooks are blunt.** Pre/post tool-call interception only. Can't reshape conversations, inject conditional system prompts, or change models mid-stream. Guardrails, not control flow.
- **Parallel agents can't safely co-edit files.** Worktrees help but add merge complexity. Current file-scope enforcement is convention-based.
- **Token spend and caching.** Long conversations are expensive. Caching works when prompt prefixes are stable — headless invocations with scoped, fixed prompts cache dramatically better than drifting interactive sessions.
- **Skills are stateless.** No memory of prior invocations. If context compacts, skill instructions may be summarized away. No checkpoint mechanism.

## Architecture direction: Option C hybrid

Interactive stages (spec authoring, review discussion) stay conversational. Deterministic stages (breakdown compilation, code review detection) run headless with scoped context. A code-driven orchestrator bridges both modes.

```
Current:
  Human -> Claude (with skill prompt) -> Claude follows instructions -> artifacts

0.5.0:
  Human -> Orchestrator (code) -> spawns scoped LLM tasks -> validates output -> advances state
                ^                                              |
                +---- config, state file, schemas -------------+
```

Skills don't disappear — they remain the human interface for interactive stages. But they stop being the orchestration layer. They become UX.

## What the orchestrator does

- Reads config to determine behavior (interactive vs autonomous, retry policy, approval mode)
- Reads/writes pipeline state per feature
- Spawns headless Claude for scoped tasks with only the context they need
- Validates output against schemas before accepting
- Handles retry with failure context
- Enforces file scope locks before spawning parallel agents
- Pauses for human input when config requires it

## Configuration layer

Config-driven behavior replaces prose-encoded decisions. Settings like approval mode, execution mode, retry policy, council composition, and detection presets become toggleable per-stage.

Presets (hotfix, autonomous-remediation, full-interactive) replace the current approach of separate prose workflows (e.g., corrective-workflow.md).

Example toggles:
- `pipeline.mode`: interactive | autonomous | supervised
- `stages.<stage>.approval`: human | auto-on-gate-pass | skip
- `stages.<stage>.execution`: headless | interactive
- `stages.<stage>.retry_policy.max_attempts`: N
- `stages.<stage>.retry_policy.strategy`: append-errors | fresh | escalate
- `pipeline.entry_point_policy`: strict | warn | permissive
- `pipeline.non_interactive`: master toggle

## State tracking

All stages track state, not just implementation. Three granularity tiers:

- **Tier 1 (stage-level):** Which stage, in-progress / gate-pending / approved. Enough for dispatch and entry point validation.
- **Tier 2 (substage):** Meaningful phases within a stage. Spec has drafting -> sections-complete -> gate-pending. Breakdown has test-compilation -> impl-compilation -> manifest-validation. Implementation has per-wave states. This is where headless orchestration operates.
- **Tier 3 (per-artifact):** Individual output validation status. Each task file, each council review. Where retry logic targets specific failures.

Design for Tier 3, implement Tier 2 first.

State file captures: current state, transition history, current task metadata (input paths, output schema, attempt count), gates passed, pending approvals. Enough for an orchestrator to cold-start from any point.

## How this addresses constraints

| Constraint | Current approach | 0.5.0 approach |
|---|---|---|
| Context rot | One long conversation accumulates everything | Scoped headless invocations with only relevant input |
| Hook limitations | Hooks try to enforce behavior from outside | Orchestrator is the control plane; hooks become safety nets |
| Parallel file conflicts | Convention-based scope declarations | Orchestrator enforces non-overlapping scopes before spawning |
| Token spend | Drifting interactive sessions break cache | Fixed prompts + scoped input = highly cacheable prefixes |
| Stateless skills | Skills re-derive state from filesystem | State-aware orchestrator fills in skill templates from current state |

## Multiple entry points

Entry points still work. The state machine supports initialization at any state. Policy (strict/warn/permissive) is config-driven. Interactive use gets warn-but-proceed. Autonomous mode enforces strict ordering.

## Dispatch scenario (autonomous operation)

A scheduled trigger could: scan a repo for failure modes, create sub-pipelines for each actionable finding, run corrective workflows headless with auto-approval, commit remediations, produce audit summaries. The state machine makes each step deterministic and recoverable. Without it, this is a prayer that the LLM follows instructions across multiple context windows.

## Failure taxonomy

Not all failures are equal — the orchestrator needs to distinguish:

- **Schema violation:** Output wrong shape. Retry with error context.
- **Semantic failure:** Valid structure but wrong content. Needs different handling (possibly different prompt, possibly escalate).
- **Resource failure:** Rate limit, timeout. Back off and retry.
- **Scope violation:** Agent edited wrong files. Reject, restore, retry.
- **Unrecoverable:** Retries exhausted. Escalate to human or skip.

## Open questions

- **Execution mechanism:** Headless Claude CLI, Agent SDK, or direct API? Each constrains tool availability, context control, and complexity differently.
- **State storage:** JSON file per feature vs. something more structured (SQLite) for concurrent pipeline support in dispatch scenarios.
- **Validation schemas:** How to formalize the implicit contracts currently embedded in skill prose and gate scripts. JSON Schema? Python validation functions? Both?
- **Interactive/autonomous boundary:** Config-driven per stage, but what's the UX for transitioning between modes mid-pipeline? (e.g., autonomous run hits a semantic failure and needs to escalate to interactive)
- **Existing gate script relationship:** Absorb into orchestrator, or keep as standalone scripts the orchestrator calls?
- **Which stages benefit most from headless execution?** Breakdown and code review are obvious. Implementation waves are likely. Spec and review are inherently interactive but should be headless-capable for autonomous mode.

## Potential next steps (when ready)

1. Inventory implicit contracts in current skill prose and gate scripts — what does each stage require as input and produce as output?
2. Prototype orchestrator for one stage (breakdown is cleanest candidate) to validate headless execution + validation + retry loop.
3. Decide on execution mechanism.

## Prior art: GSD

GSD (gsd-build) is the most mature open-source precedent for code-driven SDL orchestration — 23k+ stars, two major generations, extensive dogfooding. Comparative analysis produced two distinct contribution sets.

Wiki reference: `llm-wiki/wiki/entities/gsd.md` and `llm-wiki/wiki/summaries/gsd-firebreak-comparative-analysis.md`.

### SDL-level mechanisms worth adopting

These directly improve the output code quality produced by the pipeline, independent of orchestration mode:

- **Must-haves framework** — every task specifies three verification tiers:
  - Truths: observable behaviors the task makes true ("POST /api/auth/login returns 201 with valid JWT")
  - Artifacts: files that must exist with real implementation (min line counts, expected exports, substantive not stub)
  - Key Links: critical wiring between artifacts (specific imports connect specific modules)
  This is more mechanical than current "tests pass + lint clean" verification. Implementation agents self-verify against Must-Haves before declaring complete. Firebreak's acceptance criteria format could extend to this structure.

- **Boundary maps between waves** — each wave declares the interfaces it produces (function signatures, exported types, endpoint handlers) and what it consumes from prior waves. Wave N+1 reads Wave N's produced interfaces as consumption contract. Forces upstream interface thinking before implementation; enables deterministic cross-wave integration verification (do the imports Wave N+1 expects actually exist in Wave N's output?). Current file-scope declarations are necessary but not sufficient for catching integration bugs.

- **Decisions register (append-only DECISIONS.md)** — architectural choices tracked with rationale, scope (arch/library/pattern/convention/api/data), and revisability (No / Yes — trigger condition). New rows supersede old ones rather than editing. Read at the start of planning/research phases. Prevents re-debate of settled choices across remediation cycles and gives retrospectives a stable reference point for what decisions drove what outcomes.

- **Research phase between spec and breakdown** — dedicated codebase and library scouting step producing:
  - Don't-Hand-Roll table: problems that look simple but have existing solutions (use library X instead)
  - Common Pitfalls section: framework-specific gotchas with warning signs
  - Relevant code: existing files, reusable patterns, integration points
  - Sources with confidence levels (Context7/WebSearch/internal docs)
  Firebreak currently goes spec-to-breakdown assuming the spec author did this implicitly. Making it explicit catches framework issues (Django, Rails, React, multiprocessing) that the Martian benchmark showed are systematically missed.

- **Gap-closure loop in verification** — when per-task verification fails, feed gap context back to implementation for bounded in-loop retry before escalating. Tighter than current 2-escalation path-to-parked. GSD's verifier identifies specific gaps (missing Truths, stub Artifacts, broken Key Links) and re-runs execution with only those gaps as context.

### Orchestration scaffolding worth adopting

These support autonomous operation without directly changing output quality:

- **Agent SDK `query()` for headless stages** — answers the execution mechanism open question above. GSD proves this works at scale for scoped headless execution with tool restrictions and budget limits. Session stream yields typed events; result message contains session ID, cost, duration, error subtype.
- **Tool scoping per stage in code** — detection agents get read-only tools (`Read, Grep, Glob, Bash`), implementation gets read-write (`Read, Write, Edit, Bash, Grep, Glob`), verification gets read-only. Enforced as `allowedTools` parameter to `query()`. LLM literally cannot call tools outside phase scope.
- **Context engine pattern** — per-stage file manifest with automatic truncation and section narrowing. Each spawned session gets only its phase-appropriate files.
- **Declarative dispatch table** — replace skill-prose routing with named `DispatchRule` data structures evaluated in order. Each rule independently testable.
- **Typed event stream** — every stage transition, agent spawn, finding verification, gate result emits a typed event. Enables audit trails, cost tracking, debugging.
- **Disk-persisted circuit breakers** — respawn cap, retry counts, rewrite limits persisted to runtime state files so they survive session restarts and compaction.
- **Gates taxonomy** — classify each gate as pre-flight (blocks entry), revision (loops back with feedback), escalation (pauses for human), or abort (terminates). Selection heuristic: pre-flight if check happens before work; revision if after; escalate when revision can't resolve; abort when continuing is dangerous.

### Firebreak mechanisms to preserve (stronger than GSD)

- Council review (6 specialized agents) vs GSD's single discuss phase
- Breakdown-gate.sh (DAG acyclicity + file scope conflicts + test-before-impl) vs GSD's plan-check
- SHA-256 test locking before implementation (GSD has no equivalent)
- Baseline snapshot for regression prevention between waves (GSD has no equivalent)
- Multi-agent adversarial detection (7 detector groups + deduplicator + challenger)
- Intent register and path tracing
- Detection presets with wave sequencing

### Honest evidence note

Neither project publishes external SDL benchmark data showing "our pipeline produces N% better code than baseline." Firebreak has internal metrics (95-98% first-attempt pass, 0 regressions across ~290 tasks, 10% introduction rate) and Martian results for the code review component only. GSD has no published metrics but its own codebases are substantial, well-engineered, dogfooded evidence.

Both pipelines are structurally plausible — gates, verification, gap closure, scope enforcement, multi-agent review. The specific structural differences above explain where each pipeline is likely stronger or weaker than the other.
