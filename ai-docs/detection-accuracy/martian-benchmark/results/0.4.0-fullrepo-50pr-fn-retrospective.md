# v0.4.0 Full-Repo Benchmark — False Negative Retrospective

**Date**: 2026-04-21
**Scope**: All 71 false-negative goldens from the 50-PR full-repo pilot (`results/judge_consensus_fullrepo_50pr.json`)
**Focus**: Recall-improvement opportunities. Precision considerations noted but not central.

> **Pilot caveat.** This analysis is built on a single benchmark cycle. Per-PR judge variance (±3-5pp across 3 Sonnet 4.6 judges) and orchestrator non-determinism mean some specific categorizations below may shift across runs. Bucket sizes are directional. Recommendations prioritize patterns likely to recur, not one-off misses.

---

## Method

For each of the 71 goldens the consensus judge marked `match: false`, I read the golden text and the corresponding review to understand *why* the match didn't happen. Goldens were clustered by pattern (shape of the issue + likely reason for the miss).

## FN distribution

| Repo | FN | % of all FN | F1 on PR set |
|------|-----|-------------|--------------|
| sentry | 21 | 30% | 29.8% |
| discourse | 15 | 21% | 42.6% |
| keycloak | 15 | 21% | 34.6% |
| cal_dot_com | 12 | 17% | 47.5% |
| grafana | 8 | 11% | 50.9% |

Sentry contributes 30% of all misses — the largest single opportunity. Keycloak's F1 drag (34.6%) is a mix of concurrency, wrong-value specifics, and framework-specific issues.

## FN buckets

### 1. Naming / typo goldens (~10 FN)

*Examples*: `stopNotificiationsText` typo, `santizeAnchors` typo, `test_from_dict_inalid_data` typo, error message wording in PR10600, Javadoc phrasing in PR37634.

**Why missed:** low-severity style. Explicitly outside the `behavioral-only` preset. The Challenger would classify these as `structural` or `info` and filter them.

**Recovery cost:** requires expanding the preset to include naming/style patterns. Historically this has hurt precision more than it has helped recall. Not recommended without a principled way to filter down to "name X is clearly wrong" from generic naming observations.

### 2. Concurrency / race conditions (~8 FN)

*Examples*: `retryCount + 1` stale read needing atomic increment (PR14943), async `findMembers` race (discourse PR8), lazy `@loaded_locales` thread-safety (discourse PR9), time-window inconsistency under clock skew (PR79265), double-checked locking with missed check + cache-error handling (PR90939 G1+G2), reader thread race in test (PR40940 G2), `sample_rate = 0.0` falsy (sentry PR3).

**Why missed:** The broadened `behavioral` definition correctly accepts concurrency-triggered bugs (per `code-review-guide.md`: "concurrent execution is normal operation, not a hypothetical change"). But the Detector isn't *systematically* generating sightings for concurrent execution paths — it reports only when the concurrent scenario jumps out from the code.

**Recovery potential:** **+5-8 TPs** via an explicit concurrency audit sub-prompt: *"For each mutation, shared-state read, or cached value the diff introduces or touches, enumerate concurrent execution scenarios and check invariants. Specifically look for: read-modify-write without atomicity, check-then-act without lock, missing double-checked locking, lazy init without memory barrier."*

Precision risk: **low-medium**. Concurrency bugs are typically real when flagged; false positives are rare because the reasoning has to be concrete.

### 3. Security patterns (~5 FN)

*Examples*: SSRF via unvalidated `open(url)` (discourse PR4 G1), postMessage `targetOrigin` should be origin not full URL (PR4 G3), `X-Frame-Options: ALLOWALL` disabling clickjacking protection (PR4 G4), `TopicEmbed` XSS via unvalidated `contents` (PR4 G5), Python built-in `hash()` non-determinism breaking cache key stability (sentry PR3 G2).

**Why missed:** The AI failure mode checklist (`fbk-sdl-workflow/ai-failure-modes.md`) doesn't enumerate specific security patterns like SSRF, clickjacking, XSS, CSRF, cryptographic non-determinism. Generic "security bypass" language exists but isn't pattern-matching.

**Recovery potential:** **+3-5 TPs** via explicit security-pattern sub-rules added to the checklist or a dedicated security-focused detector pass. Patterns to enumerate: unvalidated URL fetches, broken origin validation, frame-ancestors / X-Frame-Options, unescaped HTML/template interpolation, predictable tokens (hash-based, signature-based), missing CSRF state.

Precision risk: **low**. Security goldens are genuinely high-severity; false positives here are also valid "leave a comment" candidates.

### 4. Wrong-value / wrong-name specifics (~15 FN)

*Examples*: alias literal `"alias"` where other call-sites use `"idp-alias-" + i` (PR32918 G2), returned default keystore provider where BouncyCastle expected (PR33832 G1), `slotStartTime` used where `slotEndTime` should be (PR8330 G1), `IS_TEAM_BILLING_ENABLED` logic inversion (PR10967 G4), `findByName(server, client.getId(), server.getId())` wrong argument order (PR36880 G2), Chinese locale with Traditional-vs-Simplified inconsistency, wrong key when updating type (sentry PR5 G2), `updateManyByCredentialId` with empty data preventing `@updatedAt` bump (PR22532 G1).

**Why missed:** these require the reviewer to *know* what the correct value should be. Conventions elsewhere in the repo help (full-repo access caught several like this already), but many require either a spec, prior-version comparison, or domain knowledge.

**Recovery potential:** **hard**. Some are tractable by stronger convention-matching ("grep for similar literals elsewhere, flag this one as different"), but the rest need external knowledge.

Precision risk: **high**. Broad "this value looks suspicious" rules generate many FPs.

### 5. Compile / static-type goldens (~3 FN)

*Examples*: `dbSession.Exec(args...)` arity mismatch (Go would not compile, PR79265 G3), `picocli.exit` → `System.exit()` (Java problematic direct exit, PR36882 G1), `res?.data` undefined because `res` is a fetch Response not a parsed object (PR11059 G5).

**Why missed:** we discovered and ran `go vet` on at least one grafana PR, but project-native compiler/typechecker execution is not consistently enforced. These bugs are deterministically visible to a real compiler.

**Recovery potential:** **+2-3 TPs** by making compile/typecheck a required tool call for full-repo reviews when the language has one (tsc, tsc-check, go build, go vet, javac, flow, pylint strict mode).

Precision risk: **near zero**. Compiler errors are ground truth.

### 6. Logic inversions (~8 FN)

*Examples*: slug gating inverted (PR10967 G4), `userIdsFromOrg` guard excluding org-level members for childless orgs (PR22345 G2), `===` on dayjs objects always false (PR8330 G2), empty-string array initial state (PR14740 G5), `findByType` returning resources from wrong scope (PR36880 G3), record_for_host case-normalization mismatch (discourse PR10 G3), regex pattern not anchored to full domain (discourse PR3 G2).

**Why missed:** when a condition flips, an operator changes, or a variable swaps, the reviewer doesn't systematically enumerate *old behavior for each input vs new behavior for each input*. The change looks structurally innocuous and the reviewer doesn't re-derive the truth table.

**Recovery potential:** **+3-5 TPs** via explicit sub-prompt: *"For each conditional whose predicate, operator, or variable changed in the diff, write out the old decision table and the new decision table, then check whether any input now produces a different outcome than the code author intended."*

Precision risk: **medium**. Decision-table enumeration produces noise on diffs with many conditional changes but the reasoning forces grounded output.

### 7. Test integrity nuances (~5 FN)

*Examples*: fixed sleep that would be flaky (sentry PR93824 G2), `time.sleep` monkeypatched earlier in file so test sync doesn't actually wait (sentry PR93824 G4), test name says X but tests Y (sentry PR77754 G3, discourse PR8 G3), HTTP method mismatch PUT vs DELETE in test (discourse PR8 G3), shared-mutable default in dataclass (sentry PR77754 G1).

**Why missed:** test files exist and are reviewed, but the reviewer tends to treat tests as source-of-truth rather than auditing them adversarially. The `test-integrity` finding type covers these but detection is weak.

**Recovery potential:** **+3-4 TPs** via explicit test audit sub-prompt: *"For each modified test: (a) does the test name describe what the test actually asserts? (b) do any mocks, monkeypatches, or fixtures invalidate the assertion (e.g., `time.sleep` patched away)? (c) is the assertion strict enough to catch the behavior it claims to check? (d) are shared mutable defaults avoided?"*

Precision risk: **low**. Test audits produce specific, falsifiable findings.

### 8. Cross-function API contract breaks (~6 FN)

*Examples*: Breaking changes in error response format (sentry PR5 G1), `queue.shutdown(immediate=False)` signature may not exist (sentry PR95633 G1), `res.data` undefined on fetch Response (PR11059 G5, also in #5), wrong key when updating type (sentry PR5 G2).

**Why missed despite full-repo:** The reviewer reads the changed file and often the immediate callers, but doesn't always trace to every call-site's expectation about shape/signature. Similar in shape to the consequence-uncertainty pattern we fixed for the Challenger, but on the Detector side.

**Recovery potential:** **+3-4 TPs** via "for every exported/public symbol the diff modifies (removes, renames, changes signature, changes return shape), enumerate callers and verify shape compatibility" sub-prompt.

Precision risk: **low**. Concrete and verifiable.

### 9. Content-review items (~3 FN)

*Examples*: Italian translation in Lithuanian locale file, Traditional Chinese in Simplified Chinese file, anchor-count mismatch in translated text.

**Why missed:** these are content/translation correctness issues, not code defects. A code reviewer is not the right tool.

**Recovery:** out of scope. Leave for translation reviewers or locale-aware tooling.

### 10. Architectural / cross-request-lifecycle (~3 FN)

*Examples*: asymmetric cache trust (grafana PR103633 G1 — cached grants trusted but cached denials re-fetched), nil request panic across multiple middleware methods (grafana PR76186 G1), traceID dropped silently after refactor (grafana PR76186 G2).

**Why missed:** these require reasoning about the full request lifecycle or architectural contract between layers, not local code semantics. The reviewer reads each file in isolation.

**Recovery potential:** **unclear**. A "describe the full request path through the diff" sub-prompt might help, but risks hallucination if the reviewer doesn't have enough context. Probably needs targeted design-review rules per-repo.

Precision risk: **medium-high**. Hard to verify architectural claims without domain knowledge.

## Ranked recall-improvement candidates

| Bucket | Est. recoverable TPs | Precision risk | Implementation difficulty |
|--------|---------------------|----------------|--------------------------|
| Compile/static-type enforcement | 2-3 | near zero | low |
| Security pattern enumeration | 3-5 | low | low |
| Test integrity audit sub-prompt | 3-4 | low | medium |
| Cross-function API trace | 3-4 | low | medium |
| Concurrency audit sub-prompt | 5-8 | low-medium | medium |
| Logic inversion branch-enum | 3-5 | medium | medium |
| Architectural / request-lifecycle | unclear | medium-high | hard |
| Wrong-value specifics | ~0 (hard) | high | hard |
| Naming / typos | 0 (scope) | high | requires preset change |

**Plausible additive lift from the first five rows: ~15-20 TPs on 136 goldens = recall lift from 47.8% → ~58-62%.**

Precision would likely *improve* slightly (fewer rejections, more verified findings), so F1 would rise from 41.3% → ~48-52% — comfortably in the leaderboard's top-10 range if that held up across multiple cycles. **Not a promise**; this is an upper-bound estimate assuming every recovery idea works as well as expected, which it won't.

## Per-repo observations

### Sentry (21 FN, 30% of all misses, F1=29.8%)

- Heavy on test integrity (flaky sleep, monkeypatched sleep, name mismatch, shared mutable dataclass default): ~6 FN
- Python-specific value traps (`hash()` non-determinism, `0.0` falsy, datetime as key): ~3 FN
- API contract breaks (queue signature, response shape, breaking changes): ~3 FN
- Cross-paginator correctness (get_item_key floor/ceil on datetime): ~2 FN
- The `order_by=-datetime` paginator interaction across PR1 G3 and PR2 G3 is the same underlying issue in two PRs — if the reviewer learned "floor/ceil on a non-numeric key is a type error" once, it would cover both.

**Sentry-focused recall work would combine test-integrity audit, Python-idiom patterns, and paginator-shape checks.** Potentially 8-10 TPs of lift.

### Keycloak (15 FN, 21%, F1=34.6%)

- Framework-specific (BouncyCastle provider, picocli exit, RuntimeException catch-width, session-vs-delegate recursion): ~6 FN
- Wrong-value specifics (alias literal, findByName args): ~3 FN
- Translation/locale content: ~4 FN (out of scope)
- Test quality (reader thread race, credential id not set): ~2 FN

The translation FNs drag keycloak's headline down ~3pp without being addressable by code-review tooling. If we exclude those, keycloak F1 ≈ 40% on actual code issues. Reporting both numbers separately would be more honest.

### Discourse (15 FN, 21%, F1=42.6%)

- Security patterns (SSRF, XSS, clickjacking, postMessage origin): ~4 FN, all in PR4 — a single "security-vulnerabilities audit" pattern would catch most
- CSS/layout issues (float+flex interaction, `-ms-align-items` never existed): ~2 FN — effectively out of scope
- Concurrency / thread-safety (findMembers async, @loaded_locales): ~2 FN
- Case/regex correctness (record_for_host lower(), domain anchor): ~2 FN
- Migration/data-layer issues: ~2 FN

### Cal.com (12 FN, 17%, F1=47.5%)

- Logic inversions (slug gating, userIdsFromOrg guard, wrong slot time var): ~3 FN
- Fetch API shape (res.data vs Response): 1 FN
- Prisma-specific (empty data object, retryCount atomic, updateMany scope): ~3 FN
- Miscellaneous: ~5

Cal.com's recall is already OK. Marginal improvements from logic-inversion and Prisma-idiom awareness.

### Grafana (8 FN, 11%, F1=50.9%)

- Concurrency (double-checked locking, CountDevices+UPSERT race): ~3 FN
- Architectural (asymmetric cache trust, nil request panic, traceID drop): ~3 FN
- Go-specific (dbSession.Exec arity): 1 FN
- Test coverage gap: 1 FN

Grafana is our strongest repo. The architectural FNs (cache asymmetry, request-lifecycle) are the hardest to address.

## What the analysis does *not* say

- The TP-recovery estimates are optimistic upper bounds. Individual prompts often under-deliver relative to analysis-level expectations.
- Per-PR variance is wide enough that some of the specific FN categorizations above would shift on a re-run.

## Recommended sequencing if/when recall work proceeds

1. **Compile/typecheck enforcement** (cheapest, lowest risk). Add a "discover and run the project's typechecker or static analysis tool" step as a required part of full-repo review. Ship separately from prompt changes; measure impact.
2. **Security pattern enumeration** in `ai-failure-modes.md`. Explicit pattern list. Low risk; additive.
3. **Test integrity audit** sub-prompt. Targeted, narrow scope.
4. **Concurrency audit** sub-prompt. Higher potential but more prompt surface to tune.
5. **Cross-function API trace** + **logic inversion branch-enum**. Combine or stage; both require the Detector to produce more sightings that the Challenger then verifies.

Each should ideally be evaluated independently (one-variable change) against the benchmark before combining. That's 5+ cycles if done rigorously. Given capacity constraints (one cycle per week at most on Max 20 plan), this is a quarter's worth of benchmark work if prioritized — not something to start before other 0.4.0 release work is complete.
