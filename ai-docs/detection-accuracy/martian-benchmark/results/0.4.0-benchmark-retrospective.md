## Inline Execution Fallbacks

5 of 21 review attempts (24%) fell back to inline execution — the orchestrator performed detection itself instead of spawning the decomposed T1 agents and Challenger pipeline.

### Failed reviews

| PR | Attempt | Cost | Duration | Findings | Notes |
|----|---------|------|----------|----------|-------|
| cal.com PR7232 | Batch 1 | $2.48 | 673s | 11 | Re-ran successfully in batch 2 ($0.90, 210s, 6 findings) |
| discourse PR5 | Batch 2 | $1.05 | 323s | 5 | SCSS mixin refactor (small diff) |
| discourse PR6 | Batch 2 | $0.79 | 271s | 6 | Ruby URI parsing |
| discourse PR8 | Batch 2 | $2.22 | 465s | 10 | Ember.js group management |
| discourse PR10 | Batch 2 | $2.54 | 657s | 14 | Rails embeddable hosts |

### Pattern

- All 4 batch-2 failures are discourse (Ruby/Ember) PRs. The 6 successful discourse reviews (PR1-4, PR7, PR9) completed normally.
- No cal.com PRs failed in batch 2 (PR7232's batch-1 failure resolved on retry).
- Inline reviews produce findings but without adversarial verification (no Challenger), so false positive suppression is absent. These results are invalid for benchmarking.
- The inline fallback text explicitly says "Detection performed inline by the orchestrator" — the detection script correctly flagged these.

### Possible causes

- **Agent spawning failure**: The orchestrator may hit a context or rate limit that prevents sub-agent creation, falling back to inline detection as a safety net.
- **Diff characteristics**: The failing discourse PRs don't share an obvious size pattern (5-14 findings, $0.79-$2.54 cost). Not correlated with diff complexity.
- **Batch timing**: All 4 failures are in batch 2 (the 16-review run). Possible that sustained API load over the batch increased spawn failure rate. The 300s inter-review delay may not be sufficient when reviews themselves take 3-10 min.

### Action items

- Re-run the 4 invalid discourse reviews
- Monitor whether the same PRs fail again (transient vs deterministic)
- If re-run succeeds: transient (likely rate/capacity related)
- If re-run fails: investigate diff-specific triggers

### Recall regression analysis

10 TPs lost, 4 gained, net -6. The losses fall into three patterns:

**Pattern 1: Judge mismatch — detection correct, framing unmatched (3 cases)**
- PR8330 G2: v0.4.0 F-01 describes the exact same dayjs `===` bug and suggests `.isSame()`. The judge didn't match because the finding title leads with "dead code" rather than "dayjs comparison."
- PR14740 G4/G5: dedup and empty-array issues are covered in F-05, F-10, F-11 but framed at the schema level rather than the runtime level the golden describes.

**Pattern 2: Demoted or reframed by pipeline (4 cases)**
- PR7232 G1: v0.4.0 detected the *consequence* (orphaned DB rows, F-01) rather than the *mechanism* (forEach async without await). Deeper framing, but unmatchable to mechanism-level golden.
- PR10600 G2: wrong error message reframed as "silent filtering without diagnostics."
- discourse PR2 G2: typo explicitly rejected as info-level nit by Challenger.
- discourse PR7 G3: CSS lightness value below behavioral detection threshold.

**Pattern 3: Genuinely not detected (3 cases)**
- PR14740 G1 (High): case-insensitive blacklist bypass — data-integrity issue outside behavioral detection targets.
- PR10967 G4 (Medium): IS_TEAM_BILLING slug logic inversion — boolean flag error not in current target set.
- discourse PR3 G2 (Medium): regex partial domain match — security issue rejected as nit during detection.

### Finding style as a judge-matching lever

The benchmark judge prompt says "accept semantic matches" and "same code location AND same behavioral consequence is a match." In practice, when v0.4.0 leads with downstream consequences ("orphaned DB rows") and the golden leads with the code mechanism ("forEach async without await"), the judge sees two different problems.

Golden comments are consistently **mechanism-first**: they name the code construct and its immediate misbehavior. v0.4.0's decomposed agents tend to lead with the **pattern label or architectural consequence**, widening the semantic gap even when detection is correct.

**Proposed experiment**: Take the 3 Pattern 1 (judge mismatch) cases, manually restyle the v0.4.0 finding descriptions to be mechanism-first, and re-run just those through the judge. If match rate improves, stylistic tuning in agent output instructions is a real recall lever that requires no detection changes.

**Implication**: Finding descriptions should lead with the specific code construct and its immediate misbehavior, then describe broader consequences. This maps to what golden authors write and what the judge prompt is calibrated to match.

## Mechanism-First Style Experiment

The instruction "In the sighting title and opening sentence, name the exact code expression that misbehaves and state what it does wrong" was added to all 5 behavioral agents (G1-G4 + IPT). Two PRs were re-run to test the effect on the 3 judge-mismatch cases.

### Results

| PR | Golden | v0.3.5 | v0.4.0 pre-style | v0.4.0 +style | |
|----|--------|--------|------------------|---------------|--|
| PR8330 | G2 (dayjs ===) | F-2 match | null | **F-1 match** | Recovered |
| PR14740 | G1 (case sensitivity) | F-9 match | null | **F-4 match** | Recovered |
| PR14740 | G4 (uniqueGuests dedup) | F-15 match | null | null | Still lost |
| PR14740 | G5 (empty string array) | F-5 match | null | null | Still lost |

**PR8330**: F1 22% → **40%** (100% recall recovered, precision improved from 14% to 25%)
**PR14740**: F1 19% → **33%** (recall 40% → 60%, precision 12% → 23%)

The style instruction recovered 2 of the 3 targeted cases. PR14740 G1 (case-sensitivity bypass) was not just a judge-matching improvement — it was a Pattern 3 "genuinely not detected" case that the mechanism-first framing caused the agent to actually find. Leading with the exact code expression pushes agents toward more concrete observation, not just better presentation.

### Why G4 and G5 are still lost

**G4 (uniqueGuests dedup)** — Finding granularity problem. The issue IS in the review, twice: F-04's body says "the existing-attendee dedup inside the same filter has the same problem in reverse — allowing duplicate attendee rows" and the intent register says "duplicates inside a single submission are silently inserted twice." But it's embedded as supporting context inside the adjacent case-sensitivity finding (F-04) rather than surfaced as its own finding. The judge matches against individual findings, not body text within adjacent findings.

**G5 (empty string array)** — Severity threshold problem. The review explicitly rejects it: "the real weakness is that it does nothing useful for the `[""]` default, which is a UX concern covered by F-13 and F-09 semantics, not a dead guard." The agent detected it, evaluated it, and intentionally triaged it out as a nit absorbed by adjacent findings.

### Two distinct output problems

1. **Out-of-charter findings leaking in** — Behavioral agents producing structural/fragile findings that inflate FP count. This is a filtering problem solvable by an orchestrator-level gate that drops findings whose type doesn't match the active preset before Challenger batching.

2. **In-charter findings getting consolidated** — Like G4's dedup issue absorbed into the case-sensitivity finding. Both are genuinely behavioral, but the agent treated them as one finding because they share a code location. This is a finding-granularity problem.

The first problem has a clean architectural solution. The second is a judgment call about when two observations at the same location are one finding vs two. The benchmark treats them as separate because the golden set has separate comments. A human reviewer might reasonably consolidate them.

The mechanism-first instruction helps indirectly with problem 2 — by forcing the agent to name the specific expression, it's harder to lump "case sensitivity bypass" and "no input dedup" into the same finding because they're different expressions doing different wrong things. But it didn't fully prevent consolidation in this case.

### Finding boundary heuristic

The golden comments across these 20 PRs follow a consistent pattern: each describes a single observable misbehavior at a specific code location. They don't reward architectural elegance, pattern violations, or defensive coding. They say "this code does X, it should do Y, here's what breaks."

The behavioral group's charter already aligns with this — it looks for code that misbehaves, not code that could be structured better. The remaining question is when to split: **if two observations share a code location but involve different expressions producing different wrong results for different inputs, they are separate findings.** This would recover G4 (dedup is a different expression from case-sensitivity, triggered by different input) without inflating output for genuinely consolidated cases.

## Minimum-2-Rounds Experiment

All 11 single-round PRs re-run with both style instruction and mandatory 2 rounds.

### Result

| Version | TP | FP | FN | P | R | F1 |
|---------|----|----|-----|---|---|-----|
| v0.3.5 baseline | 44 | 160 | 15 | 21.6% | 74.6% | 33.5% |
| v0.4.0 (original) | 38 | 113 | 21 | 25.2% | 64.4% | **36.2%** |
| v0.4.0 +style +min2 | 41 | 152 | 18 | 21.2% | 69.5% | 32.5% |

The min2 rule recovered 3 TPs but added 39 FPs. F1 dropped from 36.2% to 32.5% — below even the baseline. The mandatory second round generates too much noise relative to the signal it recovers. **The original respawn-gating logic outperforms a blanket minimum.**

Decision: **Revert the min2 rule.** Keep the style instruction (mechanism-first framing), which showed clear value on the targeted re-runs without inflating FP count.

### Key insight: output volume is the differentiator

The #1 tool averages **3.3 comments/PR** with 56% signal. Firebreak averages **7.6 comments/PR** with 25% signal. We produce more than double the output at half the signal ratio.

Every high-ranking tool follows the same pattern: **fewer, higher-confidence findings.** Tools that produce more comments (cubic-dev at 6.9, copilot at 5.2, coderabbit at 4.4) consistently rank lower despite comparable recall.

Firebreak's recall (64.4%) already exceeds every tool on the leaderboard except cubic-dev (75.9%, which has 30% precision). The problem is entirely precision — we detect the right things but emit too much alongside them.

### Methodology note

All benchmark tools had **full repository access** (forked PRs with complete codebase). Firebreak ran **diff-only**. Claude Code raw (37.6% F1) had full repo access and no special prompts — just its natural GitHub review behavior. Firebreak nearly matches it despite the context disadvantage.

### Strategic implication

The competitive lever is not detecting more — it's **emitting less**. The Challenger should be filtering more aggressively, and the orchestrator-level type filter for out-of-charter findings is critical. Targeting ~3-4 comments/PR with >45% signal ratio would place Firebreak in the top 5.
