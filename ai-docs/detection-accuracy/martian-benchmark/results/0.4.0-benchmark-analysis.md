# v0.4.0 Detector Decomposition — Benchmark Analysis

**Date**: 2026-04-11 through 2026-04-13
**Scope**: 20 PRs (10 cal.com/TypeScript, 10 discourse/Ruby) from Martian Code Review Benchmark
**Baseline**: Firebreak v0.3.5 monolithic detector, 50-PR consensus-judged run (2026-04-09)
**Judge**: Consensus protocol (3x independent Opus sub-agents, majority vote) for all evaluations

---

## 1. What We Tested

The v0.4.0 rearchitecture decomposed the monolithic code review detector into 7 domain-specific Tier 1 detector groups, a Sighting Deduplicator, and batched Challengers. The behavioral-only preset spawns Groups 1-4 (value-abstraction, dead-code, signal-loss, behavioral-drift) plus an Intent Path Tracer.

We ran a series of benchmark experiments to measure whether decomposition improved detection accuracy, and iterated on several instruction-level changes to address problems we found.

## 3. Experiments and Changes (Chronological)

### 3.1 Initial v0.4.0 Run (20 PRs)
- **Result**: P=25.2%, R=64.4%, F1=36.2%
- **Finding**: Recall competitive (64.4% vs 68.4% baseline) but precision only marginally improved (25.2% vs 20.5%). Average 7.5 findings/PR — still too many.
- **Inline fallbacks**: 5 of 21 attempts (24%) fell back to inline execution. Discourse PR5 required 3 attempts. Identified as transient API/capacity issue.

### 3.2 Mechanism-First Style Instruction
- **Hypothesis**: v0.4.0 agents lead with consequences ("orphaned DB rows") while golden comments lead with mechanisms ("forEach async without await"). The judge can't match different framings.
- **Change**: "In the sighting title and opening sentence, name the exact code expression that misbehaves and state what it does wrong."
- **Result on 2 targeted PRs**: Recovered 2 of 3 judge mismatches. PR8330 F1 22%→40%, PR14740 F1 19%→33%. One recovery (case-sensitivity bypass) was a genuine detection improvement, not just a framing fix.
- **Assessment**: Valuable instruction. Kept.

### 3.3 Minimum 2 Rounds
- **Hypothesis**: Single-round reviews miss golden comments because agents don't get a second look.
- **Change**: All agents run minimum 2 rounds regardless of round 1 output.
- **Result on 11 PRs**: Recovered 3 TPs but added 39 FPs. F1 dropped from 36.2% to 32.5%.
- **Assessment**: The respawn-gating logic was already making the right cost/benefit tradeoff. **Reverted.**

### 3.4 Leaderboard Analysis
- **Finding**: Top tools produce ~3-4 findings/PR at 45-55% signal ratio. Firebreak produces ~7.5/PR at 25%. The competitive lever is emitting less, not detecting more.
- **Cubic uses confidence scoring** (1-10) with a threshold filter. Their unfiltered variant (10.4/PR, F1=43.2%) and filtered variant (3.5/PR, F1=61.8%) use the same engine — the threshold alone is worth +18.6pp F1.
- **Devin hides findings** behind "View N additional findings" — surfaces ~2/PR from ~7.4 detected.
- **All benchmark tools had full repo access.** Firebreak ran diff-only.

### 3.5 Confidence Scoring
- **Change**: Detector self-scores confidence (1-10). Deduplicator tracks agent count. Challenger assigns confirmed/weakened. Orchestrator computes final score via deterministic script: `self_score × 1.2^(agents-1) × challenger_multiplier`, threshold at 6.0 (later raised to 8.0).
- **Result**: At threshold 6.0, no findings were filtered — scores clustered at 9-10. At threshold 8.0, only 6 findings filtered across 20 PRs. Self-assessed confidence does not discriminate between TPs and FPs (both cluster high).
- **Assessment**: Useful for observability. Not an effective precision lever with current score distributions.

### 3.6 Charter Filter
- **Hypothesis**: Behavioral-only preset produces structural/test-integrity/fragile findings that the benchmark can't score. Dropping them improves precision without meaningful recall loss.
- **Implementation**: `charter-filter.sh` — deterministic script checks finding type against preset's allowed types.
- **Post-hoc result on original v0.4.0 detection**: -53 FP, -6 TP. P=34.8%, R=54.2%, **F1=42.4%**. Best overall result.
- **Assessment**: Highest-leverage single change. The 6 lost TPs were findings our agents detected correctly but classified as the wrong type, which led to the next change.

### 3.7 Consequence-Based Type Classification
- **Hypothesis**: Agents classify by pattern shape ("naming issue → structural") rather than consequence ("naming issue that breaks runtime → behavioral"). This causes the charter filter to drop legitimate behavioral findings.
- **Change**: Type definitions rewritten — `behavioral` = wrong output, data loss, or runtime failure for reachable input; `structural` = maintainability without runtime consequence; etc.
- **Result**: Cal.com improved (F1 33.3%→44.0%). Discourse regressed (F1 40.0%→30.4%). Net: no improvement due to detection variance on discourse PRs.
- **Assessment**: Directionally correct but unmeasurable against run-to-run variance.

### 3.8 Script Path Resolution
- **Issue**: Benchmark headless mode runs from a different working directory. `scripts/charter-filter.sh` not found.
- **Fix**: `S=$([ -f scripts/charter-filter.sh ] && echo scripts || echo ~/.claude/scripts)` — checks project-local first, falls back to user-global.
- **Assessment**: Infrastructure fix, not a detection change.

## 4. Key Findings

### 4.1 The decomposition didn't improve detection quality
The monolithic v0.3.5 detector found more TPs per PR than the decomposed v0.4.0 agents. Splitting detection into 4 groups + IPT added orchestration overhead (deduplication, batched challengers, respawn gating) without improving what matters — finding the right bugs. Every instruction-level change we made was recovery work addressing problems the decomposition introduced.

### 4.2 The charter filter is the one genuine insight
Dropping non-behavioral findings from behavioral-only runs provably improves precision (+9.6pp) with acceptable recall cost (-10.2pp), netting +6.2pp F1. This is a 3-line script independent of the decomposition architecture — it could be applied to v0.3.5.

### 4.3 Output volume is the primary precision problem
The Martian leaderboard shows a clear pattern: fewer, higher-confidence findings wins. Cubic (#1, F1=61.8%) averages 3.3 findings/PR at 56% signal ratio. Firebreak averages 5.5-7.5/PR at 25-30% signal. Our recall is competitive with top tools; our precision is not.

### 4.4 Self-assessed confidence doesn't discriminate
LLM self-scored confidence clusters at 9-10 for both TPs and FPs. The multi-agent agreement multiplier shows a gradient (1 agent: 12% TP rate, 4 agents: 33%) but isn't strong enough to drive filtering decisions alone. Severity is a stronger discriminator: critical findings are 100% TPs, minor findings are 10%.

### 4.5 Run-to-run variance dominates instruction-level changes
The same PR produces wildly different results across runs with identical configurations. PR4 went from 4 TP to 1 TP between runs. This variance is larger than the improvements from any single instruction change, making it difficult to attribute gains to specific interventions.

### 4.6 The benchmark comparison isn't apples-to-apples
All 38 tools on the Martian benchmark had full repository access (forked PRs with complete codebase). Firebreak ran diff-only. This suppresses recall on extra-diff-dependent findings (~21 of 43 baseline FNs are extra-diff dependent). The 42.4% F1 from charter-filtered results would likely improve further with repo context.

## 5. What's Worth Keeping

Regardless of whether the decomposition ships:

1. **Charter filter script** (`scripts/charter-filter.sh`) — deterministic type-based output filtering per preset. Applicable to any architecture.
2. **Confidence scoring script** (`scripts/confidence.sh`) — observability tool for tracking detection quality across runs. Not an effective filter yet, but useful diagnostics.
3. **Mechanism-first style instruction** — "name the exact code expression that misbehaves and state what it does wrong." Improved both detection (found issues the old framing missed) and judge matching.
4. **Consequence-based type classification** — classify by runtime consequence, not pattern shape. Correct principle even if the measurement was noisy.
5. **Benchmark infrastructure improvements** — script path resolution, inline fallback detection, review backup process, multi-version comparison tooling.

---

## 6. What Didn't Work

1. **Minimum 2 rounds** — added 39 FPs for 3 TPs recovered. Reverted.
2. **Confidence threshold as precision filter** — scores cluster too high to discriminate. Needs fundamentally different calibration or a non-self-assessed signal.
3. **Multi-agent decomposition for detection** — added complexity (dedup, batched challengers, respawn gating, inline fallbacks) without improving TP count. The monolithic detector was more reliable.

## 7. Cost

| Version | Avg cost/review | Avg duration | Projected 50-PR cost |

| v0.3.5 baseline | $2.66 | 1002s | $117 |
| v0.4.0 original | $1.78 | 457s | $89 |

The original v0.4.0 was 29% cheaper and 57% faster than baseline due to parallelized agents and behavioral-only preset. The FINAL run (with confidence scoring and additional instruction overhead) cost more than baseline. The cost advantage evaporated as we added instruction complexity.

---

## 8. Recommendation

The v0.4.0 detector decomposition does not justify release based on these benchmark results. The proven improvements (charter filter, mechanism-first framing, consequence-based classification) are instruction-level changes that can be applied to the v0.3.5 monolithic architecture without the decomposition overhead.

If revisiting this work:
- Apply the charter filter to v0.3.5 and benchmark that combination
- Investigate severity-based filtering (major+ only) as the primary precision lever
- Address run-to-run variance before making further instruction changes — the noise floor is too high to measure incremental improvements on 20 PRs
- Consider a 50-PR run for any change that shows promise on 20, to reduce variance
