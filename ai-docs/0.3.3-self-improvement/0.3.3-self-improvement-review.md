Perspectives: Architecture, Quality, Pragmatism

# 0.3.3 Self-Improvement — Spec Review

## Architectural Soundness

### R-01: Category-to-type migration unspecified [blocking]

The spec introduces a two-axis classification (type + severity) but does not account for migrating the existing `category` field and its values (`semantic-drift`, `structural`, `test-integrity`, `nit`). The sighting format template, finding format template, and Category Values section in `code-review-guide.md` all use the old single-axis system. The Detector and Challenger agent definitions reference the old category field.

Three assets need migration: `code-review-guide.md` (sighting format, finding format, category values section), `fbk-code-review-detector.md` (category output), `fbk-code-review-challenger.md` (category validation, nit-downgrade behavior).

The Challenger's "downgrade to nit" behavior conflicts with the spec's statement that nits are excluded from the type axis. The spec must clarify: does the Challenger reject sightings that would have been nits, or does it mark them as nits in a separate channel?

**Resolution:** Add ACs for migrating existing format templates from `category` to `type + severity`. Add AC for canonical classification definition in `code-review-guide.md`. Clarify Challenger nit-downgrade behavior under the new system.

**Convergence:** All three agents independently identified this as blocking. (Architect #1, Guardian #4, Builder #7)

### R-02: Linter relocation handoff undefined [blocking]

AC-09 removes Bash from the Detector and relocates linter execution to SKILL.md, but the spec does not define how linter output enters the Detector's context. No schema, format, size constraints, or usage instructions are specified. Should linter output be treated as pre-formed sightings, supplementary context, or scope hints? Should `linter` be added as a detection source value alongside `spec-ac`, `checklist`, `structural-target`?

**Resolution:** Define a convention for how linter results are injected into the Detector spawn prompt. State the intended use (supplementary context, not pre-formed sightings). Add `linter` as a detection source value if linter output generates attributable findings.

### R-03: Type axis overlap needs disambiguation rules [important]

The four type categories (behavioral, structural, test-integrity, fragile) have ambiguous boundaries. Sentinel confusion is `behavioral` (wrong results) or `fragile` (future breakage)? Dead infrastructure is `structural` (organization) or `fragile` (maintenance risk)? Without decision rules, Detectors classify inconsistently across units and Challengers reclassify unpredictably.

**Resolution:** Add a disambiguation rule to the canonical definitions: "If the code produces wrong results now, it is `behavioral`. If it produces correct results but will break under realistic changes, it is `fragile`. If it is a code organization problem with no correctness risk, it is `structural`."

### R-04: Missing AC for canonical classification definitions [important]

The canonical type and severity definitions in `code-review-guide.md` are the single source of truth that Detector, Challenger, and `existing-code-review.md` all reference. No AC verifies this definition was implemented.

**Resolution:** Add AC verifying `code-review-guide.md` contains type axis definitions (behavioral, structural, test-integrity, fragile) and severity axis definitions (critical, major, minor, info).

### R-05: Adjacent observation destination unspecified [important]

AC-13 adds an "adjacent observation" channel to the Challenger, but the spec does not state where adjacent observations flow. The SKILL.md orchestration loop has no handler for a third output type beyond verified findings and rejections. Do they surface to the user? Feed the next Detector round? Append to the retrospective?

**Resolution:** State the downstream destination. Recommended: adjacent observations are appended to the retrospective as informational items, not fed back into the detection loop.

### R-06: `quality-detection.md` path inconsistency [important]

The spec references `quality-detection.md` with implicit `fbk-sdl-workflow/` path context, but the actual file is at `fbk-docs/fbk-design-guidelines/quality-detection.md`. If an implementing agent takes the path literally, it will create content in the wrong location or introduce broken references.

**Resolution:** Add a file path map at the top of the spec listing every target file with its full relative path. (Architect #3, Guardian #2, Builder #1 all flagged this.)

### R-07: AC numbering gaps [informational]

AC numbering has gaps (no AC-04, AC-19, AC-20, AC-30) from rejected proposals. Not architecturally significant but may confuse implementing agents.

**Resolution:** Add a note explaining the numbering convention (AC numbers correspond to original proposal IDs; gaps indicate rejected proposals).

### R-08: Convention visibility [informational]

Two conventions in existing assets that the spec does not mention: (1) Agent definition frontmatter (`name`, `description`, `tools`, `model`) — the Detector's `description` field should be updated to reflect the expanded output schema. (2) Detection source values are canonically defined in `code-review-guide.md` line 21, not invented in SKILL.md.

## Testing Strategy

### R-09: Confirmed test break not identified [blocking]

`test-code-review-structural.sh` Test 5 asserts Bash is in the Detector tools list. AC-09 removes Bash. This test will break. The spec's "Existing tests impacted" section does not identify this.

Additionally, Test 24 checks for specific keyword strings including `magic number` and `test name`. AC-36 expands "magic numbers" to "bare literals" and AC-38 expands "test name contradictions" to "non-enforcing tests". If implementation replaces rather than appends, Test 24's keyword greps will fail. The spec's "may need updated threshold" assessment misidentifies the break risk — the count threshold is safe; the keyword checks are at risk.

**Resolution:** Add Test 5 and Test 24 to the impacted tests section with specific required changes.

### R-10: 30+ ACs with no proposed test coverage [blocking]

The testing strategy proposes 5 structural tests covering 17 ACs. The remaining 28+ ACs have no proposed tests. These cover `implementation-guide.md`, `task-compilation.md`, `test-authoring.md`, Challenger, SKILL.md, `code-review-guide.md`, and `existing-code-review.md`. All are structural grep checks — the same pattern as the 5 proposed tests. The gap is an authoring omission, not an untestability issue.

**Resolution:** Add structural tests for all AC groups, or explicitly document which groups are intentionally left uncovered and why. (Guardian #6, test reviewer defect 5)

### R-11: Two-axis classification needs cross-file consistency tests [important]

A single-file grep can confirm `type` and `severity` appear in the Detector definition, but cannot verify the allowed values match the canonical definition in `code-review-guide.md`. An implementer could write old category values in the Detector while new values are in the guide, and the structural test would pass.

**Resolution:** Add cross-file consistency tests that extract type/severity values from `code-review-guide.md` and verify the same values appear in Detector and Challenger definitions. (Guardian #3)

### R-12: UV steps have no test mapping [important]

UV-1 through UV-4 are listed with no corresponding test and no override declaration. These describe AI agent runtime behavior that cannot be tested by structural greps.

**Resolution:** Add explicit "Behavior verified by manual QA step" override for each UV step, with structural proxy tests where feasible (e.g., SKILL.md contains cross-unit deduplication instruction as proxy for UV-3). (Test reviewer defect 2)

### R-13: Integration seams have no test coverage [important]

All four declared integration seams have no proposed tests and no override invocation. Additionally, undeclared seams exist: `existing-code-review.md` severity ordering depends on `code-review-guide.md` definitions (AC-48), and the Challenger references `code-review-guide.md` canonical definitions at runtime.

**Resolution:** Add structural proxy tests for terminology consistency across seam boundaries, or invoke "Seam not testable in current infrastructure" override. Declare the additional seams. (Test reviewer defects 3, 4)

### R-14: Vague test descriptions — expected counts not specified [important]

Proposed count tests state "matches expected total after additions" without naming the target number. Current `ai-failure-modes.md` has 5 items + 7 additions = 12. Current `quality-detection.md` has 6 targets + 5 additions = 11. State the expected counts so implementing agents have deterministic targets. (Test reviewer defect 7)

### R-15: AC-09 test is absence-only [important]

AC-09's proposed test checks Bash is absent from Detector tools but has no positive assertion that linter execution was added to SKILL.md. The test should be paired: Bash absent from Detector AND linter execution present in SKILL.md. (Test reviewer defect 1)

### R-16: Test reviewer AC placement too shallow [informational]

The proposed keyword test for AC-49 through AC-54 confirms words appear in `fbk-test-reviewer.md` but cannot verify Tier 1 items are in the Tier 1 section (not Tier 2) or checkpoint items are at the correct checkpoint.

**Resolution:** Structural tests should verify section placement, not just keyword presence. (Guardian #5)

## Over-engineering / Pragmatism

### R-17: Spec scope is large but should not be split [informational]

50 ACs across 11 files. About 35 ACs are "add a paragraph to a markdown file" — Haiku-level work. The genuinely complex ACs are concentrated in the Detector/Challenger/SKILL.md/code-review-guide cluster (~20 ACs). Splitting would create artificial seam management between specs sharing the same output schema definitions. Builder estimates ~22-24 tasks, 4 waves, max width 7-8. Feasible in a single pass.

**Recommendation:** Do not split. Batch pure-additive files into early waves; leave schema-touching files for later waves. (Builder #2, #6)

### R-18: Two-axis classification is justified [informational]

The two-axis system is the industry standard (SonarQube, CodeClimate, CodeRabbit). The implementation cost is bounded: two fields replacing one in sighting/finding schemas. The previous single-dimension system had a real problem — 60 findings with no way to sort by urgency. This is not over-engineering. (Builder #3)

### R-19: `code-review-guide.md` density risk [important]

This file is 94 lines and receives 8 ACs, roughly doubling its size. At ~180-200 lines it enters the zone where agents start skimming.

**Resolution:** Mandate that new sections are added as clearly headed subsections under the existing structure, not appended as a wall of text. (Builder #5)

### R-20: Cross-file duplication is correct for agent-consumed instructions [informational]

The coordinated changes (dead infrastructure in 4 files, string-based classification in 3 files) are correctly designed. Each file is consumed by a different agent at a different pipeline stage, and agents do not share context. Canonical definitions in `code-review-guide.md` with contextually appropriate instructions in each consuming file is the right pattern. (Builder #4)

## Threat Model Determination

**Security-relevant characteristics:** This feature modifies instruction files that guide AI agent behavior. No data is touched, no trust boundaries are crossed, no entry points are added, no auth/access control changes are made. The Detector tools change (removing Bash) reduces the attack surface by removing execution capability from an analysis agent.

**Decision: No.** No new trust boundaries, no data handling changes, no external API interaction. Removing Bash from the Detector reduces attack surface. No threat model artifact needed.
